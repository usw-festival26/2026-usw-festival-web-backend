# AGENTS.md

## 1. 프로젝트 개요

- 프로젝트 성격: `Spring Boot 3.4.x`, `Java 21`, `Gradle` 기반 백엔드
- 주요 기술 스택:
  - `spring-boot-starter-web`
  - `spring-boot-starter-validation`
  - `spring-boot-starter-data-jpa`
  - `PostgreSQL`
  - `Spring Boot Actuator`
- 기본 설정 파일: `src/main/resources/application.yml`
- 데이터베이스 연결은 환경 변수 기반이며, `SUPABASE_DB_*` 계열 값을 사용한다.

## 2. 작업 원칙

- 변경은 항상 작은 작업 단위로 나눈다.
- 한 번에 하나의 목적만 해결한다. 기능 추가, 리팩터링, 설정 변경을 한 커밋에 섞지 않는다.
- 요구사항이 불명확하면 과도한 추론보다 현재 코드베이스에 자연스럽게 맞는 최소 구현을 우선한다.
- 기존 코드 스타일과 구조가 생기기 전까지는 단순하고 일관된 구조를 유지한다.
- 사용하지 않는 추상화, 과도한 공통화, 선제적 최적화는 피한다.

## 3. 디렉터리 및 코드 구조 가이드

초기 단계에서는 아래 방향을 기본값으로 삼는다.

- `controller`: HTTP 요청/응답 처리
- `service`: 비즈니스 로직
- `domain` 또는 `entity`: JPA 엔티티
- `repository`: 데이터 접근
- `dto`: 요청/응답 DTO
- `config`: 설정 클래스

구조를 새로 도입할 때는 다음을 지킨다.

- 패키지명은 역할이 드러나게 작성한다.
- API 스펙용 DTO와 영속 엔티티를 직접 혼용하지 않는다.
- 컨트롤러에서는 비즈니스 로직을 최소화한다.
- 검증은 가능하면 요청 DTO에서 `javax/jakarta validation`으로 우선 처리한다.

## 4. 구현 규칙

- Java 버전은 `21`을 기준으로 한다.
- Spring Boot 관례를 우선하고, 별도 프레임워크나 라이브러리 도입은 필요성이 명확할 때만 진행한다.
- 환경 변수, 비밀번호, 토큰, DB 접속 정보는 코드에 하드코딩하지 않는다.
- 설정값이 추가되면 `application.yml` 또는 `@ConfigurationProperties` 기반으로 관리하는 것을 우선 검토한다.
- 예외 처리는 향후 공통화할 수 있지만, 초기에는 무리하게 전역 구조를 만들기보다 필요한 범위에서 명확하게 처리한다.
- 로그는 디버깅에 필요한 정보만 남기고 민감 정보는 출력하지 않는다.

## 5. 데이터베이스 및 JPA 주의사항

현재 `application.yml` 기준 주요 사항:

- DB는 PostgreSQL 계열을 사용한다.
- `ddl-auto: update` 상태이므로, 스키마 변경이 실제 DB에 어떤 영향을 주는지 항상 의식하고 작업한다.
- `open-in-view: false` 이므로, 지연 로딩 의존 코드는 서비스 계층에서 트랜잭션 범위를 분명히 해야 한다.
- 엔티티 설계 시 양방향 연관관계는 꼭 필요할 때만 사용한다.
- N+1 가능성이 보이면 조회 메서드, fetch 전략, 쿼리 방식까지 함께 점검한다.

## 6. 검증 및 실행

작업 후 기본 검증은 아래 순서를 권장한다.

1. `./gradlew test`
2. 필요 시 `./gradlew bootRun`

아직 테스트 기반이 약한 초기 프로젝트이므로, 기능을 추가했다면 가능하면 해당 변경을 보호하는 테스트도 함께 추가한다.

검증 관련 원칙:

- 빌드/테스트를 돌리지 못했다면 이유를 작업 내용에 명시한다.
- 환경 변수 의존으로 실행이 어려운 경우, 어떤 값이 필요한지 함께 남긴다.
- 단순 문서 수정이 아닌 코드 변경이면 최소한 테스트 실행 여부는 확인한다.

## 7. Git 및 커밋 규칙

- 커밋은 반드시 작업 단위별로 진행한다.
- 커밋 메시지는 반드시 한국어로 작성한다.
- 커밋 메시지는 반드시 `'{type}: {한글 커밋메세지}'` 형식을 지킨다.
- 의미 없는 중간 커밋보다는, 되돌리기 쉽고 리뷰 가능한 단위로 커밋한다.
- 사용자나 다른 작업자가 만든 변경사항은 임의로 되돌리지 않는다.
- 브랜치/PR 작업 시, 무엇을 왜 바꿨는지 한 줄 요약이 가능해야 한다.

예시:

- `feat: 회원 엔티티 및 기본 리포지토리 추가`
- `feat: 축제 행사 조회 API 초안 구현`
- `chore: Supabase DB 연결 설정 보완`

## 8. PR 및 작업 기록

PR을 작성할 때는 `.github/PULL_REQUEST_TEMPLATE.md`를 따른다.

특히 아래 내용을 분명히 적는다.

- 구현한 기능
- 변경 이유
- 테스트 여부 및 결과
- 리뷰어가 보면 좋은 참고 사항

## 9. 문서 업데이트 원칙

아래 상황에서는 이 `AGENTS.md`도 함께 갱신하는 것을 권장한다.

- 프로젝트 구조에 팀 규칙이 생겼을 때
- 예외 처리, 로깅, 테스트 전략이 정해졌을 때
- 패키지 구조 또는 계층 규칙이 확정됐을 때
- DB 마이그레이션 방식이 바뀌었을 때
- 배포/운영 절차가 추가됐을 때

- 예외 응답 포맷
